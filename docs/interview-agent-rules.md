# 面试空间 Agent 叙事规则（66 条）

> 来源：穿刺 `interview-story(1).html`（1192 行，8 Slides + 5 Overlays）提取的叙事逻辑、内容结构、时间线处理、视觉原则。
> 每一条规则必须被 Agent 在生成面试演示时遵守。

---

## 零、核心事实约束（最高优先级）

### 规则 0：叙事编排不得篡改事实

本规则优先级高于所有其他规则。任何叙事编排、结构优化、故事提炼，**不得以歪曲事实为代价**。

**禁止行为：**

1. **禁止捏造事实** — 不得凭空编造不存在的公司、职位、项目、数字、客户名称、技术栈。所有声称的经历、日期、公司、职位、量化指标，必须可追溯到 `ResumeData` 中的对应字段。如果简历里没写"服务 100+ 研发用户"，就不能生成这个数字。

2. **禁止混淆经历归属** — 不得将 A 公司的经历嫁接到 B 公司、将早期经历的时间线挪到后期、将不同阶段做的事混为一谈。每段经历的公司、职位、时间区间必须与简历数据严格一致。

3. **禁止因果倒置** — 不得将事后才形成的认知，写成驱动早期决策的原因。如果简历中某项洞察的 `storyReflection` 明确发生在某段经历之后，该洞察不能作为该段经历之前或之中的战略判断出现。使用"事后总结"是诚实的——事后合理化是不诚实的。

4. **禁止挂羊头卖狗肉** — 不得将常规执行包装为战略突破、将参与者角色包装为负责人、将协助的工作包装为独立成果。职位的实际权责范围不得被夸大。如果一个项目你是参与者而非 owner，叙事应如实反映参与程度。

5. **禁止数字注水** — 不得将"效率提升 20%"改为"效率提升 50%+"、将"参与 3 个项目"改为"主导 10+ 项目"。数字可以从 highlights 中提取，但提取出来的数字不能被放大或修饰。

**允许的叙事加工（在此范围内是安全的）：**

- 重新排列呈现顺序（如将最具代表性的经历放在 Slide 4）
- 压缩/省略次要细节（如将 4 段实习选 3 段展示）
- 提炼抽象主题（如从具体工作中归纳出"平台架构认知"这样的元能力标签）
- 使用精确的时间格式（如将"2023 年"格式化为 `2023.06`，前提是简历中有对应年月）
- 应用本规范定义的叙事结构、视觉规则和信息密度控制

**判断标准：**

> 如果简历的作者本人看到生成的面试演示，他是否会对任何一条陈述感到"这不是我做的"或"这时间不对"或"这个数字我没说过"？如果会——那就是违规。

## 一、叙事弧线设计（10 条）

### 规则 1：八页固定故事线

面试演示必须是 8 页，不可多不可少。8 页对应面试 15-20 分钟的信息承载量。页序固定：

```
Hero → Foundation → Gap → Platform Build → Agent Leap → Full Stack → Impact → The Arc
```

对应叙事函数：**你是谁 → 从哪来 → 看到了什么问题 → 你做了什么基础建设 → 你的核心突破是什么 → 全貌是怎样的 → 带来了什么结果 → 回到起点，前后对比**

### 规则 2：每页必须有时间锚点

每页左上角 eyebrow label（`.eb`）必须包含时间区间。格式：`{英文阶段名} · {年份区间}`。

| Slide | Eyebrow 模板 |
|-------|-------------|
| 1 | 无（Hero 页例外，不放时间） |
| 2 | `Foundation · {最早年份}—{最晚年份}` |
| 3 | `The Gap · {入职行业年份}—{关键突破年份}` |
| 4 | `Platform Build · {公司名} Phase 1 · {年份区间}` |
| 5 | `Phase 2 · {公司名} {当前职位} · {年份区间}` |
| 6 | `Full Stack`（无年份，这是能力全景图） |
| 7 | `Impact`（无年份，这是结果总结） |
| 8 | `The Arc`（无年份，这是未来展望） |

### 规则 3：首尾必须形成视觉呼应

Slide 3 和 Slide 8 必须使用同一张结构图，但数据状态不同：
- **Slide 3**：Before 状态——展示行业现状（国外工具垄断、零 AI、无协同）
- **Slide 8**：After 状态——展示你的成果（国产覆盖、Agent 增强、云原生）
- 两张图的结构（节点位置、连线方式、图层关系）完全一致，只在颜色和标注上变化

### 规则 4：叙事由"问题发现"驱动，不是"经历罗列"

任何经历在出场之前，必须先建立"为什么这段经历值得讲"。具体做法：
1. 先展示行业问题空间（垄断/低效/数据孤岛）
2. 再引入个人经历（"在这个背景下，我做了什么"）
3. 最后提炼发现（"这几个发现定义了后续所有产品方向"）

经历本身不是故事——经历回答了某个行业问题的过程才是故事。

### 规则 5：每页必须有一个收束句

每页（除 Hero 和 The Arc）的最后一段必须是 card-accent（带左边框的强调卡片），内容为该页的核心洞察：
- Slide 2：实习经历的可迁移性声明
- Slide 3：两个工具验证的原则
- Slide 4：关键架构决策的 WHY
- Slide 5：产品战略一句话
- Slide 6：行业覆盖标签
- Slide 7：成长弧线 + 引用

### 规则 6：全篇第一人称"我"

所有正文使用"我"视角。不使用"该员工"、"本人"、"他"。不使用第三人称描述自己的工作。引用和反思语句尤其要口语化。

### 规则 7：先战略决策，后执行细节

每个阶段的叙述顺序是：
1. 先陈述你做的**战略判断**（"我的核心判断是……"）
2. 再展开**执行工作**（象限/卡片/列表）
3. 最后收束**验证结果**（"这一决策让……"）

不允许先列工作内容再补一句"这体现了我的战略思维"。战略思维必须在行动之前出现。

### 规则 8：最后一页必须形成闭环

Slide 8 不是"总结"，是"回到起点"。具体做法：
- 复用 Slide 3 的行业全景图，更新为 After 状态
- 用 Before/After 对比卡片显式展示变化
- 最后一句是个人品牌陈述（见规则 50）
- 底部一行总结人生阶段："{N} 年，{M} 个阶段：{阶段主题}。每一段经历都是下一段的前提。"

### 规则 9：信息密度梯度——Slide 5 是密度峰值

8 页的信息量不是均匀分布的。密度曲线如下：
- Slide 1-2：低密度（建立信任）
- Slide 3：中密度（建立问题空间）
- Slide 4：中密度（展开工作）
- **Slide 5：最高密度**（核心展示，允许 2-3 倍于其他页的内容量）
- Slide 6：中高密度（全景图）
- Slide 7：中密度（结果）
- Slide 8：低密度（收束，留白）

Slide 5 标 ★CORE，是面试官注意力峰值，内容必须最丰富。

### 规则 10：叙事节奏——三阶段递进

全文隐含三段式结构：
- **阶段一（Slide 1-3）**：建立可信度——我是谁、从哪来、看到了什么
- **阶段二（Slide 4-6）**：展示能力——我做了什么、怎么做的、为什么这样做
- **阶段三（Slide 7-8）**：呈现结果与愿景——带来了什么、这意味着什么

三个阶段的分界点分别是入职当前公司（Slide 4）和成果展示（Slide 7）。

## 二、Slide 1 Hero 页规则（5 条）

### 规则 11：Hero 不用照片，用能力架构图

Hero 右半部分必须是你的核心能力/产品架构图（SVG），不是头像。这张图直接告诉面试官"你做了什么"。架构图分三层：
1. **顶层**：你构建的智能/产品层（如 5 个 Agent）
2. **中层**：数据/知识底座（如 RAG 知识库）
3. **底层**：你负责的工具/平台（如一体化工具链）

三层之间有箭头表示数据流/调用关系。

### 规则 12：姓名+职位下方必须有一行职责描述

紧接姓名的不是"产品总监"（头衔），而是一行描述你实际职责的句子。格式：

```
负责{领域}的{核心工作}、{差异化能力}与{商业目标}。
管理{N}人{团队类型}，覆盖{行业1}、{行业2}、{行业3}等方向。
```

### 规则 13：Hero 核心指标区展示 4-5 个精确数字

在姓名/职责下方放置一个带边框的信息卡片。内容为 4-5 行关键指标：
- 每行包含至少一个**精确数字**（加粗着色）
- 数字类型覆盖：Agent/产品数量、效率提升百分比、用户/客户数量、覆盖率
- 百分比用 `50%+`、`20%+` 格式，不用 `约 50%` 或 `50%-60%`
- 行与行之间用换行分隔，不用 bullet

### 规则 14：底部放权威背书标签

Hero 页底部放 1-2 个权威背书的 compact 标签（pills）。格式：
```
{权威机构/项目名称}（如"工信部汽车工具链摸底调研"）
```
标签用小号字体 + 边框 + 品牌色（violet/gold）。

### 规则 15：架构图的每个子模块必须有标签

架构图中的每个方块/节点包含：模块名称 + 一行说明文字。节点之间用箭头连接，箭头标注数据流方向。图层之间用不同底色区分（如 Agent 层=violet 底色，工具链层=gold 底色）。

## 三、Slide 2 Foundation —— 实习弱化原则（8 条）

### 规则 16：所有早期经历强制压缩到一页

无论简历中有多少段实习/初级经历，在面试演示中全部压缩到 Slide 2 的单页内。最多展示 3 段——超出 3 段的选最有代表性的 3 段。

### 规则 17：实习页标题使用"基础训练"框架

标题不能是"工作经历"或"职业经历"。格式：
```
{时间跨度}前，我在{行业/领域}完成了{专业方向}的基础训练
```

正文导语说明这些经历的价值是**方法论**而非**领域知识**。使用句式："这段经历定义了我的产品方法论：先理解业务本质，再定义产品形态，用数据验证决策。"

### 规则 18：每段实习标准卡片格式

每段实习一张卡片（`.fd-card`），包含：
1. **公司名**（加粗，大字）
2. **职位名称**（不用"实习生"后缀，用正式职位名如"后台产品"）
3. **时间区间**（精确到月，如 `2020.12—2021.03`）
4. **3-4 条交付物**（每条以 `·` 开头，用动宾结构，如"测评中台从 0 到 1 建设"）
5. **一条"学会了"收尾**（格式见规则 19）

### 规则 19：每条"学会了"必须提炼可迁移的元能力

每张实习卡片底部用 `▶ 学会：{元能力}` 收尾。元能力必须是抽象能力而非具体技能：

| ✅ 正确 | ❌ 错误 |
|--------|--------|
| 从零搭建一个业务系统 | 学会了用 Axure |
| B 端平台架构 + 数据驱动 | 写了 SQL 报表 |
| 用户增长 + 商业化闭环 | 做了营销活动 |

元能力需要能直接连接到后续职业选择。

### 规则 20：页末必须声明可迁移性

Slide 2 的最后用 card-accent（violet 色）明确声明：

"这几段经历给我的不是某一个行业的领域知识——而是一套跨领域可迁移的产品方法论。{方法论要点 A}让我理解{后续能力 A}，{方法论要点 B}让我学会{后续能力 B}，{方法论要点 C}让我建立{后续能力 C}。后来在面对{后续陌生领域的技术名词}时，这套方法论让我能快速切入业务本质，而不是停留在功能表面。"

关键：必须列举 3 个后续面对的**具体陌生技术名词**（来自后续行业的真实术语），证明方法论确实迁移了。

### 规则 21：实习不单独标注"实习"标签

职位名称中使用正式职位名（"后台产品"而非"后台产品实习生"），时间区间已经说明了这是早期阶段。页面整体框架（"基础训练"标题）已经暗示了这是学习阶段。

### 规则 22：三段实习之间必须有递进关系

三段经历的选择和排列必须体现能力递进：
1. 第一段：建立**平台架构认知**（B 端系统从 0 到 1）
2. 第二段：建立**数据驱动能力**（报表体系 + B 端架构）
3. 第三段：建立**商业思维**（增长 + 商业化）

排列按时间顺序，但选择标准是"这三段分别教会了你什么不同的事"。

### 规则 23：实习经历不能超过总页数的 12.5%

1/8 页是实习的篇幅上限。任何早期经历（实习、第一份工作不满 1 年）不能跨页。如果你有 5 年工作经验，早期 1.5 年只能占 1 页。这是刻意的比例控制——面试官更关心你最近做了什么。

## 四、Slide 3 Gap/Tension —— 行业问题空间（6 条）

### 规则 24：Slide 3 必须先展示行业全景图，再引入个人经历

页面布局顺序：
1. Eyebrow 标签（The Gap · 年份区间）
2. 标题：一句关于行业现状的判断（"我进入了{行业}——看到了一个被{问题描述}的领域"）
3. **行业全景图**（SVG 信息图，最大视觉元素）
4. 一段小字说明/数据补充
5. **个人经历嵌入**（Timeline 或卡片格式）
6. 三个发现
7. Card-accent 收束

全景图必须在个人经历之前。全景图没有"我"——它是客观行业事实。

### 规则 25：行业全景图必须标注现有方案的状态

V 模型/市场地图中的每个节点需标注：
- **工具/方案名称**（具体产品名，如 "IBM DOORS"、"PREEvision"、"CANoe"）
- **状态标记**（国外垄断/国产可用/空白）
- **关键属性**（桌面单机/Web 协同/AI 能力）

标注用颜色编码：
- 国外垄断：灰色/黑色弱化（`rgba(0,0,0,0.02)` 底色）
- 国产已有：teal/green 高亮 + `国产 ✓` badge
- AI 能力：violet 标记

### 规则 26：个人经历用 Timeline 格式嵌入

在全景图下方，以 Timeline 格式展示你在该阶段的个人经历。Timeline 格式要求：
- 左侧竖线（1.5px，浅色）
- 每个节点：圆点（金色）+ 日期 + 事件描述
- 每行一个关键事件，按时间顺序排列
- 关键术语用金色加粗（`<span class="accent">`）
- 最后一行是成果节点（如"两个工具交付两家头部主机厂"）

### 规则 27：三个发现用编号列表，每条有因果逻辑

Timeline 下方放一个 card-accent，列出 3 条发现。格式：
```
{N} 个发现定义了后续所有产品方向：
① {现象} → {根因} → 需要{解决方案}
② {现象} → {根因} → 需要{解决方案}
③ 事后总结：{做完之后的洞察}——这后来成了{后续工作的基础}
```

第三条如果是在做完之后才意识到的，加"事后总结"前缀。保持时序诚实。

### 规则 28：行业术语保持原样，不加解释

汽车电子领域的专业术语（CAN 总线、UDS 诊断、SOME/IP、AUTOSAR AP、DDS、ARXML、ROS2）直接使用，不在正文中加括号解释。面试演示的目标读者（面试官）应该能理解这些术语。如果面试官不理解，这正好创造解释机会。

### 规则 29：Slide 3 必须有一个"仿真验证"节点

在 V 模型的测试域右臂，必须包含一个"仿真验证"节点（区别于传统的部件测试→验收测试序列）。它是你后续服务仿真平台的故事锚点。Slide 3 的仿真节点用 cyan 色标记。

## 五、Slide 4 Platform Build —— 平台建设（5 条）

### 规则 30：标题陈述战略决策

Slide 4 的标题不是"我在{公司}的工作"，而是一个**战略决策陈述**：

```
我选择先做{决策 A}，再做{决策 B}——{决策理由的一句话总结}
```

例如："我选择先建平台底座，再做上层模块——平台层优先是架构设计的硬道理"。

### 规则 31：工作内容用象限布局，非列表

工作内容使用 2×2 四象限网格（`.quad`），不是 bullet list。每个象限：
- 编号（01/02/03/04，暗示优先级）
- 独立顶部色条（3px，不同颜色）
- 标题（方向名称，如"SOME/IP 通信工具 0→1"）
- 描述（3-4 句，包含客户/场景/成果）
- 颜色编码：blue=通信工具、teal=平台基建、rose=低代码、gold=售前

### 规则 32：必须有一个 card-accent 解释 WHY

象限下方放一个 card-accent，解释核心架构决策的动机：

"关键架构决策：{决策内容}。{决策描述}——这些不是'通用功能'，而是让{N}个{模块类型}运行在同一个{基础设施}上的基础设施。这一决策让后续{效果}，也为{更远的工作}打下了{基础}——但当时我的目标很明确：{当时的目标}。"

格式关键：先陈述决策，再解释价值，最后诚实说明当时的认知范围（不是事后合理化）。

### 规则 33：底部标注职业晋升

页末放一个 phase badge + 说明，标注在此阶段内的晋升：
```
Phase 1 · {晋升前职位}→{晋升后职位}
{晋升时间和理由的一句话说明}。
```

### 规则 34：Slide 4 的 Overlay 是平台分层架构

平台详情 Overlay（`ov-platform`）展示五层架构：
1. 应用模块层（6 个设计模块）
2. 平台服务层（协同编辑、变更审批、版本管理……）
3. 认证与项目管理层（用户、角色、权限、多租户）
4. 统一数据模型层（跨模块关联、三级模型、Schema）
5. 存储与基础设施层（数据库、缓存、容器编排）

每层一色，底部一行总结：平台层优先的战略价值。

## 六、Slide 5 Agent Leap ★CORE —— 核心突破页（6 条）

### 规则 35：开篇用反直觉陈述

Slide 5 的第一句话不能是"我做了 Agent"，而必须是：

```
晋升后我做的第一件事不是{最酷的那个工作}——是先把{前提条件}做完整。
```

为什么？因为这句反直觉陈述让面试官产生疑问（"为什么不做 Agent？"），然后你再用 card-accent 解释原因——展示你的战略思考过程。

### 规则 36：必须解释"为什么是这个顺序"

反直觉陈述后紧跟一个 card-accent（violet），显式回答"为什么是这个顺序"：

"为什么是这个顺序？{你要做的酷事}的能力上限不取决于{酷事本身}，取决于它能调用什么{工具/数据/平台}。没有完整的{前置条件}，{酷事}就没有可操作的{基础}；没有统一的{数据层}，{检索系统}检索到的就是碎片化的数据——而不是可关联、可追溯的{知识类型}。**先有{前置条件}，再有{酷事}。**"

这段解释是面试演示中最重要的战略论述。

### 规则 37：Agent 卡片统一格式（5 个）

每个 Agent 用独立卡片展示，5 张卡片并排。卡片标准结构：
1. **Icon**：16×16 SVG 图标（放在 24×24 圆形容器中，背景为对应色 10% 透明度）
2. **名称**："{功能名} Agent"
3. **场景**：一句话说明 Agent 做什么（9-10 字）
4. **量化指标**：一个关键数字（金色加粗）

5 张卡片的顶部色条各用不同颜色：
- 文档生成：violet
- 模型生成：blue
- 一致性校验：teal
- 方案推荐：gold
- 知识检索：rose

### 规则 38：Agent 工作流用可展开详情，不占主空间

受控式 Agent 工作流（任务规划→工具调用→多步执行→状态管理→人工确认→结果归档）作为虚线边框的紧凑触发区（`.ov-trigger`），不展开在主页面上。6 个步骤的详细说明放在 overlay 中。

### 规则 39：RAG 知识库必须分层展示

RAG 知识库在主页面用紧凑四段式数据流展示（数据层→处理层→检索层→应用层），详细架构放在 overlay。主页面展示 RAG 的唯一目的是证明：**你有完整的数据底座思维，不是套壳 LLM**。

### 规则 40：页末产品战略一句话

Slide 5 最后用 card-accent 收束为核心产品战略声明：

"产品战略：不做 {热门概念的套壳}。{你的核心能力} 的有效性取决于 {数据/平台的某一层} 的质量，{这一层} 的质量取决于 {更底层的工作} 的程度。**没有 A → 没有 B → C 只是 D。**"

用层层递推的因果链表达战略深度，最后一句话用加粗。

## 七、Slide 6 Full Stack —— 全生命周期（3 条）

### 规则 41：Pipeline SVG 覆盖全生命周期

全生命周期 Pipeline 图（SVG）必须包含以下结构：
1. **顶部**：5 个生命周期阶段标签（架构设计→服务编排→仿真验证→DevOps→观测运维）
2. **第二层**："AI Agent + RAG 智能层——贯穿研发全生命周期"（横贯全图）
3. **主体**：5 个阶段的独立卡片（不同颜色），每个标注：
   - 阶段名称 + 简要描述
   - 核心功能（2-3 行小字）
   - 该阶段对应的 Agent（violet 小字）
4. **底部**：RAG 知识库底座（虚线框），标注"各阶段数据沉淀→向量嵌入→语义检索→反哺 Agent"
5. **最底部**：如果有跨行业延伸，加一行（如"具身智能延伸"）

### 规则 42：每个生命周期阶段标注对应的 Agent

每个阶段卡片中必须标注该阶段使用哪个 Agent。Agent 标注用 violet 小字，不抢阶段本身的视觉权重。这个标注的目的是证明：Agent 不是独立功能，而是嵌入研发生命周期。

### 规则 43：页末放行业覆盖标签

Pipeline 下方放一行 domain tags（pills），展示覆盖的行业方向：
```
汽车 · 东风等主机厂 | 军工 · 602 所航空 | 具身智能
```
每个标签用带边框的圆角 pill，不同颜色边框对应不同行业。

## 八、Slide 7 Impact —— 成果展示（4 条）

### 规则 44：成果用三张指标卡，一张一维度

三张卡片覆盖三个不同维度：
1. **客户/规模维度**（gold 顶部色条）：客户数量、用户数量、覆盖节点
2. **效率/质量维度**（violet 顶部色条）：效率提升百分比、质量改善
3. **行业/广度维度**（teal 顶部色条）：行业覆盖、权威认可

每张卡片结构：大数字（32px，加粗 900）+ 标签（12-13px 加粗）+ 4-5 行说明文字。
三个维度之间不重复、不交叉。

### 规则 45：指标卡之后放成长弧线

三张卡片下方放一个 card-accent（violet），展示个人成长弧线：

"我的成长弧线：{年份} {公司} → {年份} {公司} → ……
{N} 年时间，经历{职位 1}→{职位 2}→{职位 3}的{晋升方式}，管理{N}人{团队类型}。"

成长弧线用年份+公司+职位三要素的线性排列，被动词/主动词随阶段演进明显变化。

### 规则 46：成果页必须有一句对面试官说的引用

成果页以一句引用（`.quote`）收束。引用格式：
- 上下各一条 1px 边框线
- 字体比正文大（clamp(12px, 1.4vw, 18px)）
- 加粗 800
- 内容是第一人称，语气是"对面试官说的"

引用必须包含：
- 你今天来面试的职位
- 你做过这个职位最核心的事
- 你知道它的上限在哪、下限在哪、怎么让客户买单

### 规则 47：数字必须精确，可有小数

所有指标数字必须是精确值，不用约数：
- ✅ `2000+`、`50%+`、`>85%`、`3.7%`
- ❌ `约 2000`、`50%-60%`、`大约 85%`

如果效率提升有多个维度，分别给出各自数字，不合并成一个范围。

## 九、Slide 8 The Arc —— 闭环收束（3 条）

### 规则 48：复用 Slide 3 的同一张结构图

Slide 8 的结构图必须与 Slide 3 完全一致（节点位置、连线、图层），但所有数据更新为 After 状态：

### 规则 49：Before/After 对比用双卡片 + 箭头

在结构图下方放置 Before/After 对比：
- 左侧卡片（rose 底色，rose 边框）：BEFORE 状态，列出 3-4 个负向指标
- 中间：SVG 箭头
- 右侧卡片（teal 底色，teal 边框）：AFTER 状态，列出 3-4 个正向指标

### 规则 50：最后一句话是个人品牌陈述

Slide 8 的最后一句（可能是引用或正文），格式：

```
"我是{姓名}。从{最早的公司/岗位}做到{现在的公司/岗位}——我做的事一直在变，但逻辑始终一致：{一句话核心信念}。这一次，我看到的是{你对目标领域的愿景}。"

{N} 年，{M} 个阶段：{阶段一主题} → {阶段二主题} → {阶段三主题}。每一段经历都是下一段的前提。
```

## 十、Overlay 通用规则（5 条）

### 规则 51：每个 Overlay 必须有三个要素

每个 Overlay 面板包含：
1. **标题**（22px，900 字重）：Overlay 主题
2. **导语**（12.5px，次级色）：一段说明上下文/必要性的文字
3. **主体信息图**（SVG）：结构化展示架构/流程/分类

不允许只有文字没有 SVG 的 Overlay。

### 规则 52：Overlay SVG 的 viewBox 标准

| Overlay | viewBox | 最大高度 |
|---------|---------|---------|
| agent-workflow | `0 0 1020 160` | 150px |
| rag-detail | `0 0 900 300` | 290px |
| conflict-types | 无 SVG（纯卡片网格） | — |
| arch-detail | `0 0 900 460` | 440px |
| platform | `0 0 800 340` | 330px |

### 规则 53：Overlay 触发区必须用虚线边框 + `↗` 标记

任何可点击展开 Overlay 的区域必须：
- 使用 `ov-trigger` class
- 如果是紧凑触发区，使用虚线边框（`border: 1px dashed`）
- 如果是卡片触发区，在右上角显示 `↗` 小标记
- 在区域内标注 `[展开详情]` 或 `[详情]`（8-9px，opacity 0.5）

### 规则 54：conflict-types Overlay 用网格而非 SVG

一致性校验 7 类冲突场景的 Overlay 不使用 SVG，而使用 2 列网格卡片。原因：冲突类型是分类知识，不是架构流程，卡片网格比 SVG 更适合查阅。

每张卡片：标题（加粗）+ 3-5 个子项。第 7 项（定义完整性校验）跨全宽。

### 规则 55：所有 Overlay SVG 的颜色必须与主页面一致

Overlay 中的颜色编码必须与主页面对应元素的颜色一致：
- Agent/智能层 → violet (`#6b5ea0`)
- 工具链/架构 → gold (`#8a5c0f` / `#a07018`)
- 国产/成功 → teal (`#1a7d62`)
- 仿真 → cyan (`#318a94`)
- DevOps → green (`#4a8a5a`)
- 警示/旧状态 → rose (`#b85d6a`)
- 信息/数据 → blue (`#3d7fb8`)

## 十一、通用叙事规则（5 条）

### 规则 56：数字出现频次——每页至少 2 个量化指标

除 Slide 8 外，每页至少出现 2 个量化数字。数字类型轮换：规模类（用户数/客户数）、效率类（提升百分比）、时间类（年数/月数）、覆盖类（行业数/模块数）。同一页不重复同一类型的数字。

### 规则 57：每页至少一个 card-accent 或 quote

Card-accent（带色左边框的强调卡片）用于传递洞察和战略判断。每页至少放置一个。如果某页没有战略判断需要表达，说明该页是纯罗列——需要重写。

### 规则 58：颜色承担语义，不是装饰

颜色在全文中有固定语义映射。不随意使用颜色：
- **gold** (`#8a5c0f`)：品牌色/强调/关键数字/商业价值
- **violet** (`#6b5ea0`)：AI/Agent/智能/战略洞察
- **teal** (`#1a7d62`)：国产替代/成功/已验证/平台化
- **rose** (`#b85d6a`)：旧状态/警示/Before/问题
- **blue** (`#3d7fb8`)：信息/数据/通信
- **cyan** (`#318a94`)：仿真/验证
- **green** (`#4a8a5a`)：DevOps/交付/工程

### 规则 59：时间表述必须精确

所有时间用精确到月的格式（如 `2023.06`、`2020.12—2021.03`），不用"2023 年中"、"2020 年底"等模糊表述。时间跨度用到月：`2022.08—2023.01` 而非 `2022.08—2023`。

### 规则 60：团队规模和管理范围必须精确

首次出现管理职责时，给出精确数字：`管理 10 人产品团队` 而非 `管理约十人团队`。如果一个阶段从独立贡献者变为管理者，在晋升标注处体现。

## 十二、Agent 生成行为约束（5 条）

### 规则 61：简历数据注入但不被简历数据结构限制

Agent 接收 ResumeData 作为输入，但**不按照简历的章节结构来组织内容**。简历是数据源，不是大纲。Agent 必须：
1. 从简历提取个人锚点（公司名、职位、日期、关键数字、技能列表）
2. 从 prompt 获取领域知识（行业全景、技术栈、工具生态、竞争格局）
3. 按本规范的 8 页叙事结构组织输出，而非按简历结构

### 规则 62：SVG 必须由 Agent 生成，不使用占位符

每个需要 SVG 的 slide 和 overlay，Agent 必须生成完整的 SVG 代码。SVG 要求：
- 内联在输出中（不用外部文件引用）
- viewBox 使用本规范规定的尺寸
- 颜色遵循规则 58 的语义映射
- 文字使用 `font-family="PingFang SC, ..."` 中文字体栈
- 所有 `text` 元素必须居中或左对齐，不使用 `dominant-baseline`

### 规则 63：缺失数据的处理方式

如果 ResumeData 缺少某页需要的数据：
- **Slide 2**（实习少）：有几段展示几段，但保持"基础训练"框架和元能力提炼
- **Slide 3**（无行业数据）：从 prompt 中获取行业全景信息，不做 V 模型可改用其他行业全景图
- **Slide 4-6**（经历不够丰富）：减少象限/card 数量，保持结构，用现有数据填充
- **Slide 7**（缺少量化成果）：从 highlights 提取量化关键词，如实展示
- 绝不编造不存在的数字或公司名

### 规则 64：输出格式 — JSON 中的 PresentationDraft 结构

Agent 最终输出必须符合 `PresentationDraft` 接口的结构，但 slides 和 overlays 的 body 字段必须包含完整内容（HTML 或 Markdown），不再是空字符串。每个 slide 的 `visualizations` 字段应包含完整 SVG 字符串。

### 规则 65：降级方案 — 规则引擎作为 Baseline

当 LLM 不可用或 token 超限时，回退到现有的 `generatePresentationDraft()` 规则引擎。规则引擎生成的演示作为"结构化摘要"，LLM 生成的演示作为"叙事作品"。两者共享相同的输出格式，确保前端渲染层不需修改。
